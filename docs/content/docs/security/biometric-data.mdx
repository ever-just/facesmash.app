## Face Descriptors Explained

A face descriptor is a **128-dimensional floating-point vector** that represents the unique geometric features of a face. It is generated by a neural network (SSD MobileNet v1 + FaceNet) running entirely in the browser.

### Example Descriptor

```json
[0.0234, -0.0891, 0.1456, -0.0312, 0.0789, ...]
// 128 numbers between approximately -0.5 and 0.5
```

### Key Properties

- **Non-reversible** — A descriptor cannot be used to reconstruct or approximate the original face image
- **Compact** — 128 float32 values = 512 bytes per descriptor
- **Deterministic** — The same face under similar conditions produces similar (not identical) descriptors
- **Discriminative** — Different faces produce measurably different descriptors

## Processing Pipeline

### 1. Face Detection

The SSD MobileNet v1 neural network detects face bounding boxes in the video frame. This runs at ~3 FPS in the browser.

- Input: Video frame (HTMLVideoElement)
- Output: Bounding box coordinates + confidence score
- Processing: Client-side only (WebGL)

### 2. Landmark Detection

68 facial landmarks are detected within the bounding box for face alignment.

- Used for: Head pose estimation, eye aspect ratio, face alignment
- Not stored: Landmarks are used only during processing

### 3. Descriptor Extraction

A FaceNet-style neural network extracts the 128-dimensional descriptor from the aligned face region.

- Input: Aligned face crop
- Output: Float32Array(128)
- This is the only data that leaves the device

### 4. Quality Analysis

Multiple quality factors are computed to ensure the capture is usable:

| Factor | How It's Measured | Threshold |
|---|---|---|
| **Overall quality** | Weighted combination of all factors | > 0.35 |
| **Lighting** | Pixel intensity distribution analysis | > 0.2 |
| **Sharpness** | Laplacian variance of face region | > 0.3 |
| **Face size** | Face area relative to frame | > 15% |
| **Head pose** | Yaw, pitch, roll from landmarks | < 30 degrees |
| **Eye aspect ratio** | Eye openness detection | > 0.2 |

## Storage

### What Is Stored

```json
{
  "user_email": "user@example.com",
  "descriptor": [0.0234, -0.0891, ...],
  "quality_score": 0.82,
  "created": "2026-03-08T00:00:00.000Z"
}
```

### What Is NOT Stored

- Raw camera frames
- Face images or crops
- Facial landmarks
- Any data that could reconstruct a face

### Database

Face descriptors are stored in PostgreSQL 16 with pgvector as `vector(128)` columns. The database is hosted on a dedicated DigitalOcean Droplet with:

- HTTPS-only API access via Caddy reverse proxy (auto-TLS)
- No public database access (localhost-only PostgreSQL)
- HNSW indexes for fast cosine similarity searches
- Regular backups

## Matching Process

When a user attempts to login:

1. Liveness detection validates a real face (blink + head pose tracking)
2. A new descriptor is extracted from the live camera feed
3. The descriptor is sent to the Hono API over HTTPS
4. The server performs a **pgvector cosine similarity search** across all stored descriptors
5. The cosine distance is converted to a similarity score: `similarity = 1 - cosine_distance`
6. If similarity exceeds the adaptive threshold (default 0.45), the user is authenticated
7. A JWT httpOnly cookie is set for session management

### Matching Guarantees

- **No raw image comparison** — Only 128-number vectors are compared
- **Client-initiated** — The browser sends the descriptor; the server never requests camera access
- **One-way** — An attacker with database access cannot reconstruct faces from descriptors

## Data Deletion

When a user deletes their account:

1. All face descriptors are permanently deleted
2. All face templates are permanently deleted
3. All face scan records are permanently deleted
4. Sign-in logs are permanently deleted
5. The user profile is permanently deleted

This is an immediate, irreversible operation — not a soft delete.
